## 1. Zielbild

Wenn der Benutzer in deiner App im Beat auf den **Play-Button** eines Prompts klickt, soll deine App **nicht direkt** den n8n-Webhook aufrufen, sondern eine **Supabase Edge Function**. Diese Edge Function arbeitet als **serverseitiger Proxy** und ruft den eigentlichen n8n-Webhook auf. Dadurch umgehst du das Browser-CORS-Problem. Supabase Edge Functions laufen unter URLs im Muster `https://<PROJECT_ID>.supabase.co/functions/v1/<function-name>`. Standardmäßig verlangen sie ein JWT; für öffentliche Webhook-/Proxy-Szenarien kann man sie beim Deploy mit `--no-verify-jwt` ohne JWT-Prüfung veröffentlichen. (<a href="https://supabase.com/docs/guides/functions/deploy?utm_source=chatgpt.com" target="_blank" rel="noopener noreferrer">Supabase</a>)

## 2. Architektur

Der Ablauf sieht so aus:

```typescript
App / Browser
   ↓ POST
Supabase Edge Function: render-beat
   ↓ POST
n8n Webhook: /webhook/RenderBeat
   ↓ Stream / Antwort
Supabase Edge Function
   ↓ Stream / Antwort
App / Browser

```

Warum Supabase hier hilft:

- Die App spricht nur noch **deine eigene Edge Function** an.

- Die Edge Function spricht **serverseitig** mit n8n.

- Damit entfällt das frühere Browser-CORS-Problem gegen `casper001.duckdns.org`.

- Die Edge Function kann die Antwort von n8n direkt an den Browser weiterreichen.

## 4. Request aus deiner App

### 4.1 HTTP-Methode

`POST`

### 4.2 Request-Body

Die App sendet JSON mit diesen Feldern:

```typescript
{
  "beat_id": "<UUID des aktuellen Beats>",
  "p_text": "",
  "prompt_id": "<UUID des gewählten Prompts>",
  "llm_model_id": "<UUID des Default-Modells>"
}

```

### 4.3 Bedeutung der Parameter

`beat_id`\
Die ID des Beats, für den der Render-Prozess laufen soll.

`p_text`\
Ein freier Zusatztext. In deinem aktuellen Fall wird bewusst ein leerer String geschickt.

`prompt_id`\
Die ID des in der Oberfläche ausgewählten Prompts.

`llm_model_id`\
Die ID des Standardmodells, das für diesen Prompt verwendet werden soll.

## 5. Verhalten der Edge Function

Die Edge Function macht im Kern vier Dinge:

1. Sie beantwortet `OPTIONS`, damit Browser-CORS sauber funktioniert.

2. Sie akzeptiert `POST`.

3. Sie liest das JSON aus dem Request.

4. Sie ruft den n8n-Webhook serverseitig mit demselben JSON auf und reicht die Antwort zurück.

## 6. Der Code der Edge Function

Das ist der Code, den du aktuell dafür verwenden kannst:

```typescript
const WEBHOOK_URL = 'https://casper001.duckdns.org/webhook/RenderBeat';

const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Methods': 'POST, OPTIONS',
  'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type',
};

Deno.serve(async (req: Request) => {
  if (req.method === 'OPTIONS') {
    return new Response(null, {
      status: 204,
      headers: corsHeaders,
    });
  }

  if (req.method !== 'POST') {
    return new Response(
      JSON.stringify({ error: 'Method not allowed' }),
      {
        status: 405,
        headers: {
          ...corsHeaders,
          'Content-Type': 'application/json',
        },
      }
    );
  }

  try {
    let body: unknown = {};

    try {
      body = await req.json();
    } catch {
      return new Response(
        JSON.stringify({ error: 'Invalid or missing JSON body' }),
        {
          status: 400,
          headers: {
            ...corsHeaders,
            'Content-Type': 'application/json',
          },
        }
      );
    }

    const webhookResp = await fetch(WEBHOOK_URL, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(body),
    });

    if (!webhookResp.ok) {
      const errorText = await webhookResp.text().catch(() => '');
      return new Response(
        JSON.stringify({
          error: `Webhook error: ${webhookResp.status}`,
          details: errorText || null,
        }),
        {
          status: webhookResp.status,
          headers: {
            ...corsHeaders,
            'Content-Type': 'application/json',
          },
        }
      );
    }

    if (!webhookResp.body) {
      const text = await webhookResp.text();
      return new Response(text, {
        status: 200,
        headers: {
          ...corsHeaders,
          'Content-Type': webhookResp.headers.get('Content-Type') || 'text/plain; charset=utf-8',
          'X-Content-Type-Options': 'nosniff',
          'Cache-Control': 'no-cache',
        },
      });
    }

    return new Response(webhookResp.body, {
      status: 200,
      headers: {
        ...corsHeaders,
        'Content-Type': webhookResp.headers.get('Content-Type') || 'text/plain; charset=utf-8',
        'X-Content-Type-Options': 'nosniff',
        'Cache-Control': 'no-cache',
      },
    });
  } catch (err) {
    return new Response(
      JSON.stringify({
        error: err instanceof Error ? err.message : String(err),
      }),
      {
        status: 500,
        headers: {
          ...corsHeaders,
          'Content-Type': 'application/json',
        },
      }
    );
  }
});

```

## 7. Was der Code konkret tut

### `WEBHOOK_URL`

Das ist der echte n8n-Zielendpunkt.

### `corsHeaders`

Diese Header erlauben Browsern den Aufruf deiner Function per `POST` und `OPTIONS`.

### `OPTIONS`-Block

Wenn der Browser vorab einen Preflight schickt, antwortet die Function mit `204 No Content`. Supabase Edge Functions unterstützen HTTP-Methoden wie `GET`, `POST` und `OPTIONS`; diese Form der Behandlung ist für browserfähige Endpunkte normal. (<a href="https://supabase.com/docs/guides/functions/development-tips?utm_source=chatgpt.com" target="_blank" rel="noopener noreferrer">Supabase</a>)

### Methodenprüfung

Alles außer `POST` wird mit `405 Method not allowed` abgewiesen.

### `req.json()`

Die Function erwartet JSON. Kommt kein gültiges JSON, liefert sie `400`.

### `fetch(WEBHOOK_URL, ...)`

Der Request wird serverseitig an n8n weitergereicht. Der Body bleibt inhaltlich gleich.

### Fehlerbehandlung für n8n

Antwortet n8n nicht mit `2xx`, gibt die Function diesen Fehler als JSON an die App zurück.

### `return new Response(webhookResp.body, ...)`

Wenn n8n einen Body liefert, wird dieser direkt an den Browser weitergereicht. Dadurch kann ein Stream prinzipiell bis in deine App durchgereicht werden.

## 8. Deploy und JWT

Supabase Edge Functions verlangen standardmäßig ein JWT. Für öffentliche Functions, etwa Webhooks oder solche Proxy-Endpunkte, kann man beim Deploy `--no-verify-jwt` verwenden. Genau dafür ist dieses Flag gedacht. (<a href="https://supabase.com/docs/guides/functions/deploy?utm_source=chatgpt.com" target="_blank" rel="noopener noreferrer">Supabase</a>)

### Wichtige Konsequenz

Wenn du möchtest, dass deine App die Function ohne Supabase-Auth-Token direkt aufruft, muss die Function **ohne JWT-Prüfung** deployed oder entsprechend konfiguriert werden. Supabase beschreibt dafür sowohl das CLI-Flag `--no-verify-jwt` als auch die Function-Konfiguration mit `verify_jwt`. (<a href="https://supabase.com/docs/guides/functions/deploy?utm_source=chatgpt.com" target="_blank" rel="noopener noreferrer">Supabase</a>)

## 9. Beispiel-Deploy

### Per CLI

```typescript
supabase functions deploy render-beat --no-verify-jwt
```

Das ist der von Supabase dokumentierte Weg für öffentliche Functions. (<a href="https://supabase.com/docs/guides/functions/deploy?utm_source=chatgpt.com" target="_blank" rel="noopener noreferrer">Supabase</a>)

### Optional in `config.toml`

## 12. Rolle von Supabase im Gesamtablauf

Supabase spielt hier nicht die Rolle des fachlichen Renderers, sondern die eines **sicheren Vermittlers**:

- Es stellt den öffentlichen Endpoint bereit.

- Es löst das Browser-CORS-Problem gegen n8n.

- Es kann Auth, Logging und spätere Zusatzlogik übernehmen.

- Es reicht die n8n-Antwort an die App weiter.

Wichtig ist: Supabase erzeugt den eigentlichen Beat-Text hier nicht. Das macht weiterhin dein n8n-Workflow.

## 13. Rolle von n8n im Gesamtablauf

n8n bleibt die Fachlogik:

- Prompt laden

- Beat-Kontext ermitteln

- Default-Modell verwenden

- LLM anstoßen

- Output erzeugen

- Output streamen oder final zurückgeben

Die Edge Function ist nur der Proxy vor diesem Workflow.

## 14. Rolle deiner App

Deine App macht drei Dinge:

1. Sie sammelt die IDs:

   - `beat_id`

   - `prompt_id`

   - `llm_model_id`

2. Sie ruft die Edge Function auf.

3. Sie liest die Antwort und schreibt den empfangenen Text in `draft_text`.

## 15. Warum der Text aktuell eher „auf einmal“ erscheint

Wenn der Text im Editor nicht chunkweise wächst, liegt das typischerweise an einem dieser Punkte:

- n8n liefert trotz Streaming-Konfiguration am Ende doch nur einen Gesamtblock

- die Edge Function reicht zwar durch, aber upstream kommt erst spät nutzbarer Text

- das Frontend liest die Antwort nicht per `response.body.getReader()`, sondern mit `response.text()` oder `response.json()`

- das Frontend liest zwar Chunks, setzt den State aber erst nach Ende der Schleife

## 16. Empfohlene Frontend-Anforderung

Damit `draft_text` sichtbar live wächst, muss das Frontend:

- die Antwort als Stream lesen

- jeden Chunk sofort dekodieren

- bei jedem Chunk den Editor-/State-Inhalt aktualisieren

- den neuen Text oben vor den bisherigen `draft_text` setzen

## 17. Zusammenfassung in einem Satz

Deine App ruft eine öffentliche Supabase Edge Function `render-beat` auf; diese Funktion leitet den Request serverseitig an den n8n-Webhook weiter und gibt dessen Antwort an die App zurück, damit CORS vermieden wird und Streaming grundsätzlich bis ins Frontend durchgereicht werden kann.
