# `ublic.get_prompt_codex_payload_for_beat(...)`

## Zweck

Erzeugt aus einer Menge von Haupteintrags-IDs und einem Beat einen JSONB-Payload für den Prompt.

Diese Funktion ist jetzt zusätzlich abgesichert gegen:

- inaktive Haupteinträge

- zeitlich ungültige Haupteinträge

## Signatur

```typescript
public.get_prompt_codex_payload_for_beat(
  p_codex_entry_ids uuid[],
  p_beat_id uuid
) returns jsonb
```

## Parameter

| Parameter           | Typ      | Bedeutung                                               |
| ------------------- | -------- | ------------------------------------------------------- |
| `p_codex_entry_ids` | `uuid[]` | Haupteinträge, aus denen ein Payload gebaut werden soll |
| `p_beat_id`         | `uuid`   | Beat-Kontext für Datumsprüfung und Details              |

## Rückgabe

Ein JSONB-Array in der Form:

```typescript
[
  {
    "codex_title": "...",
    "codex_text": "...",
    "codex_details": [...]
  }
]
```

## Wie die Funktion arbeitet

### Schritt 1

Sie holt den Beat-Zeitpunkt (`effective_at`).

### Schritt 2

Sie expandiert die Eingabe-IDs.

### Schritt 3

Sie ruft `get_matching_codex_details_for_beat(...)` auf und gruppiert die Details.

### Schritt 4

Sie lädt die Haupteinträge, aber nur wenn:

- `is_active = true`&#x20;

- `valid_from / valid_to` zum Beat passen&#x20;

### Schritt 5

Sie baut den JSONB-Payload.

## Aufrufvarianten

### Einfacher Aufruf

```typescript
select public.get_prompt_codex_payload_for_beat(
  array[
    '11111111-1111-1111-1111-111111111111'::uuid
  ],
  '18f58f5a-a454-4cdc-8a60-7eec80c1b007'::uuid
);
```

### Mit mehreren IDs

```typescript
select public.get_prompt_codex_payload_for_beat(
  array[
    '11111111-1111-1111-1111-111111111111'::uuid,
    '22222222-2222-2222-2222-222222222222'::uuid
  ],

```

## `CREATE OR REPLACE FUNCTION`

```typescript
create or replace function public.get_prompt_codex_payload_for_beat(
  p_codex_entry_ids uuid[],
  p_beat_id uuid
)
returns jsonb
language sql
stable
as $function$
with target as (
  select
    s.starts_at as effective_at
  from public.beats b
  join public.scenes s
    on s.id = b.scene_id
  where b.id = p_beat_id
),
input_ids as (
  select distinct unnest(coalesce(p_codex_entry_ids, '{}'::uuid[])) as codex_entry_id
),
matched_details as (
  select *
  from public.get_matching_codex_details_for_beat(
    p_codex_entry_ids,
    p_beat_id
  )
),
grouped_details as (
  select
    md.codex_entry_id,
    jsonb_agg(
      md.codex_text
      order by md.sort_order, md.title
    ) as codex_details
  from matched_details md
  group by md.codex_entry_id
),
final_rows as (
  select
    ce.id as codex_entry_id,
    ce.name as codex_title,
    ce.description as codex_text,
    coalesce(gd.codex_details, '[]'::jsonb) as codex_details
  from input_ids i
  join public.codex_entries ce
    on ce.id = i.codex_entry_id
  left join grouped_details gd
    on gd.codex_entry_id = ce.id
  join target t
    on true
  where ce.is_active = true
    and (
      t.effective_at is null
      or (
        (ce.valid_from is null or ce.valid_from <= t.effective_at)
        and (ce.valid_to is null or ce.valid_to >= t.effective_at)
      )
    )
)
select coalesce(
  jsonb_agg(
    jsonb_build_object(
      'codex_title', fr.codex_title,
      'codex_text', fr.codex_text,
      'codex_details', fr.codex_details
    )
    order by fr.codex_title
  ),
  '[]'::jsonb
)
from final_rows fr;
$function$;
```

## Limitierungen und kritische Punkte

- &#x20;Die Funktion prüft keine `track_mentions` und kein `ai_context_mode`; sie geht davon aus, dass die IDs schon fachlich sinnvoll vorgefiltert sind.&#x20;

- &#x20;Sie ist robust gegen ungültige Haupteinträge, aber nicht gegen fachlich „unerwartete“ Roh-IDs, solange diese aktiv und zeitlich gültig sind.&#x20;

- &#x20;Die Sortierung des Outputs erfolgt nach `codex_title`, nicht nach Textfundstelle.&#x20;

- &#x20;Sie verwendet `description`, nicht `overview`.
